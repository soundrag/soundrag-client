### 2️⃣ 위치에 따라 달라지는 소리: 공간 음향 구현하기

> 스피커와 리스너의 위치에 따라 소리가 어떻게 들리는지를 구현하기 위해서 **공간 음향**이라는 개념이 중요하다고 생각합니다. **공간 음향**이란 소리의 방향과 거리에 따라 소리가 어떻게 변하는지를 나타내는 것으로 이 프로젝트는 웹 브라우저에 내장된 **Web Audio API**를 활용하여 공간 음향과 관련된 세 가지 원리를 적용한 음향 변화를 구현하였습니다.

1. 전반적인 공간 음향: **HRTF (Head-Related Transfer Function)**

   **Web Audio API** 에는 사용자와 스피커의 위치를 기반으로 전반적인 공간 음향 특성을 제공하는 `PannerNode`가 있습니다. `PannerNode`에서는 공간 음향을 구현하기 위해 선택할 수 있는 두 가지 옵션이 있습니다.
   <br>

   - **Equalpower**

   - 간단한 스테레오 효과를 구현하기 위해 사용되며 주로 2D 오디오 구현에 사용됩니다.

   - 좌, 우 채널 간의 볼륨 변화만 구현이 가능합니다.

   - **HRTF**

     - **HRTF**란 사용자의 머리를 중심으로 3D 공간에서의 음향 전달 방식을 수학적으로 모델링한 함수입니다.

     - 게임, 가상 현실에서 공간 음향을 구현하기 위해 사용되며 주로 **3D** 오디오 구현에 사용됩니다.

     - 위, 아래, 앞, 뒤 등 **다양한 방향**에서의 볼륨 변화 구현이 가능합니다.

     <br>

   이 프로젝트에서는 3D 공간 음향 구현에 주로 활용되는 **HRTF**가 적합하다고 결정하였습니다. 다양한 방향에서 음향 변화를 전달할 수 있는 **HRTF** 모델은 3D 공간에서의 스피커와 리스너 사이의 `[x, y, z]` 좌표 거리 차이를 기반으로 **소리의 감쇠효과**를 구현합니다. 이 효과를 활용하여 각 모델의 `[x, y, z]` 좌표 값을 **실시간**으로 추적하여 아래와 같이 리스너로부터 스피커의 좌표 거리가 멀어지면 볼륨을 **작게** 조절하고 가까워지면 볼륨을 **크게** 조절할 수 있었습니다.

   <br>

2. 스피커의 좌, 우 움직임: **IID(Interaural Intensity Difference)**<br>

   > 3D 공간 음향에서 소리의 좌, 우 방향성에 대한 미묘한 차이를 효과적으로 전달하기 위해서는 HRTF 뿐만 아니라 **IID**의 적용이 필요하다고 생각했습니다. **IID** 현상은 소리가 두 귀에 도달할 때의 음파의 에너지 수준 차이를 나타내는 현상으로 스피커가 왼쪽에 위치하면 왼쪽 귀에는 직접적인 소리가 더 강하게 도달하고, 오른쪽 귀에는 머리가 막고 있어 소리가 약해지는 상황을 의미합니다.

   이 프로젝트에서는 **IID** 현상을 구현하기 위해 리스너로부터 스피커의 왼쪽, 오른쪽 위치를 파악하기 위해 삼각함수 중 하나인 **cos** 함수를 사용하였습니다. **cos** 함수를 통해 스피커와 리스너의 x축, z축의 거리 차이로 각도를 계산하고 **좌, 우 위치**와 관련된 값을 설정하였습니다.

    <br>

   | 값  | 위치   | cos        |
   | --- | ------ | ---------- |
   | -1  | 왼쪽   | cos (180°) |
   | 0   | 중앙   | cos (90°)  |
   | 1   | 오른쪽 | cos (0°)   |

   _각도에 따른 **cos** 함수 계산 값_

   <br>

   **Web Audio API**의 `StereoPannerNode` 를 활용하여 아래와 같이 계산된 값이 `-1`에 가까우면 **왼쪽** 볼륨을, `1`에 가까우면 **오른쪽** 볼륨을 강조하여 좌, 우 방향성에 대한 음향 변화를 구현할 수 있었습니다.

3.  스피커의 수직 움직임: 음파의 **흡수, 반사** <br>

    > 3D 오디오 환경을 효과적으로 구현하기 위해, 단순한 볼륨 감쇠효과를 넘어서 음파의 **흡수, 반사** 원리를 활용하여 **고주파**와 **저주파** 음역대의 차이를 명확하게 조절하는 접근을 선택했습니다. 아래와 같이 **고주파** 음역대와 **저주파** 음역대와 음파의 **흡수, 반사** 원리와의 연관성, 그리고 특징을 활용하면 사용자에게 보다 입체적인 음향 변화를 제공할 수 있다고 판단하였습니다.

    <br>

    - 고주파 음역대

      - 파장이 짧아 푹신푹신한 표면에서 쉽게 **흡수, 반사**됩니다.

      - 소리의 특정 파트(목소리, 심벌즈 등)를 더 효과적으로 전달합니다.

    - 저주파 음역대

      - 파장이 길어 **흡수, 반사**가 적게 일어나며, 소리가 더 멀리까지 전달됩니다.

      - 깊은 베이스와 같은 소리를 더욱 효과적으로 전달합니다.

    <br>

    이 프로젝트에서는 **고주파** 음역대와 **저주파** 음역대의 차이를 보여주기 위해 스피커의 수직적 움직임을 천장과 바닥으로 제한하여 사용자에게 **직관적인 음향 변화**를 제공합니다. **Web Audio API**의 `BiquadFilterNode`를 활용하여 만약 사용자가 스피커를 천장으로 배치한다면 **고주파** 음역대를 **강조**하였고 스피커를 **바닥**으로 배치한다면 **저주파** 음역대를 강조하도록 구현하였습니다. 아래 그림은 스피커의 위치에 따라 음파의 흡수, 반사 작용을 시각화하고 프로젝트에 적용한 시각 자료입니다.

### 2️⃣ 저장: 저장 방식 추상화

<br>

#### (1) 저장 버튼 누르는 걸 깜빡했어요

> 사용자가 배치한 스피커와 리스너 위치 정보를 저장 버튼을 클릭하여 저장할 수 있지만 실수로 클릭하지 않는다면 동일한 설정을 재현하기 어려운 상황이 발생합니다. 이 문제를 해결하고자 **자동 저장** 기능을 통해 사용자는 별도의 저장 버튼을 클릭하지 않아도 변경된 위치 정보가 **자동으로 저장**될 수 있도록 구현하였습니다.

사용자는 버튼을 클릭하여 저장하는 방식과 달리 **아래의 조건**이 성립될 때 **자동 저장**할 수 있습니다.

- 사용자가 스피커와 리스너의 **위치나 회전 각도를 조정할 때**, 위치나 회전 값이 이전 값과 달라졌을 때 자동 저장합니다.

- 이미 저장된 위치 정보들과 새로 조정한 위치 정보가 **일치하지 않을 때**, 중복 저장을 방지하기 위해 기존에 저장된 데이터와 비교하여 새로운 위치 정보만 저장합니다.

- **5초** 동안 사용자가 위치나 회전 각도를 **변경하지 않을 때**, 사용자의 조작이 끝났음을 판단하기 위해 대기 시간을 두고 자동 저장합니다.

#### (2) 사용자가 조정한 스피커와 리스너의 위치 정보는 어디에 저장하나요?

> 로그인한 사용자는 위치 정보를 **서버**에 저장하고 로그인을 하지 않은 사용자는 **로컬스토리지**를 통해 위치 정보를 저장할 수 있습니다.

<br>

|      | 서버                                   | 로컬스토리지                         |
| ---- | -------------------------------------- | ------------------------------------ |
| 장점 | 데이터 보존성 강화<br>다중 기기 동기화 | 빠른 접근 속도<br>오프라인 사용 가능 |

<br>

위와 같이 서버와 로컬스토리지의 장점을 비교해 봤을 때 서비스를 이용하는 사용자의 성격에 따라 위치 정보 저장 방식도 구분해야 한다고 생각하였습니다.

<br>

- 로그인한 사용자

  - 개인 계정을 통해 **다양한 기기**에서 위치 정보를 활용하는 상황을 기대

  - 다른 기기에서 사용하더라도 데이터가 손실될 위험이 적은 **서버**

- 로그인을 하지 않은 사용자

  - 별도 로그인 없이 서비스를 이용하면서 위치 정보에 **빠르게** 접근하는 상황을 기대

  - 빠른 접근 속도와 온라인이 필요한 로그인 상황에 구애받지 않고 활용할 수 있는 **로컬스토리지**

<br>

이러한 방식으로 로그인한 사용자와 로그인하지 않은 사용자 모두가 기능을 이용할 수 있도록 구현하여 **사용자 접근성**을 향상시켰습니다.

<br>

#### (3) 로그인을 하지 않았던 사용자가 중간에 로그인을 시도한다면?

> 이 프로젝트에서는 로그인을 하지 않은 사용자는 스피커와 리스너의 위치 정보를 **로컬스토리지**에서 관리하고 로그인에 성공할 경우 **서버**로 업데이트합니다.

그러나 이 과정에서 **로컬스토리지**의 위치 정보와 **서버**에 저장된 위치 정보 간에 **충돌**이 발생할 수 있습니다. 이러한 **충돌**을 해결하기 위해 **낙관적 동시성 제어(Optimistic Concurrency Control)** 개념을 활용하여 아래와 같이 구현하였습니다. **낙관적 동시성 제어**란 사용자가 저장하는 데이터의 정확성을 유지하도록 제어하는 것을 의미합니다.

<br>

- 서버에 저장되는 스피커와 리스너의 위치 정보에는 반드시 **버전** 필드를 포함합니다.

- 로컬 스토리지에 저장된 위치 정보와 서버에 저장된 위치 정보 중 **일치**하는 것이 있는지 확인합니다.

  - 일치하는 데이터가 있을 경우 업데이트를 **거부**하여 데이터 충돌을 방지합니다.

  - 일치하는 데이터가 없을 경우 서버의 **최신 버전**으로 위치 정보를 **업데이트**합니다.
