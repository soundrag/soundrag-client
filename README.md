### 시스템 아키텍처

1. 사용자는 3D 공간에서 스피커의 위치를 조작할 수 있습니다.

2. 사용자는 조작한 위치에 따라 소리의 변화를 느낄 수 있습니다.

3. 사용자는 원하는 위치 정보를 저장할 수 있습니다.

   - Google 로그인한 사용자는 **서버**에 위치 정보를 저장합니다.
   - 로그인하지 않거나 오프라인 사용자는 **로컬스토리지**에 위치 정보를 저장합니다.
   - 로그인하지 않거나 오프라인 사용자가 로그인을 하면 **로컬스토리지**에 저장한 위치 정보가 **서버**에 **업데이트**됩니다.

4. 사용자는 저장된 위치 정보를 불러올 수 있습니다.

## 자세히 살펴보기

### 조작에 따라 달라지는 위치: 2D 화면에서 3D 조작하기

> 2D 화면인 모니터에서 3D 모델의 위치를 조작하려면 평면에서 **3D 좌표의 거리감**을 파악하는 것이 중요하다고 생각합니다. <br>
> 이 프로젝트에서는 변환된 2D 컴퓨터 화면 상의 좌표를 활용하여 거리감을 계산하고 위치를 파악하는 방식으로 구현하였습니다. <br>

1. 사용자가 마우스를 클릭하거나 이동할 때, 2D 화면 상의 [x, y]를 캡처하여 **NDC(Normalized Device Coordinates)** 좌표로 변환합니다.
    <img src="https://github.com/user-attachments/assets/635258b9-4f44-436d-82e6-560064e78394">

   - **NDC** 좌표란 화면의 모든 좌표를 -1에서 1 사이의 값으로 변환(정규화)한 좌표를 의미합니다.

2. 변환된 **NDC** 좌표를 기반으로, 카메라에서 시작해 마우스 위치를 향하는 **광선(Raycaster)** 을 생성합니다.
    <img src="https://github.com/user-attachments/assets/e5eec1f2-107e-4303-ab48-b5142484ba00">


3. 생성된 광선이 3D 공간 내의 가상 화면과 교차하는 지점을 찾고 **교차점까지의 깊이와 거리**를 계산합니다.

<br>

✔️ **2D 화면**에서 정규화된 좌표 값으로 위치를 계산하고 계산된 값으로 **3D 공간**에서의 움직임을 구현하였습니다.

<br>
<br>

### 위치에 따라 달라지는 소리: 공간 음향 구현하기

> 스피커와 청취자의 위치에 따라 소리가 어떻게 들리는지를 구현하기 위해서 **공간 음향**이라는 개념이 중요하다고 생각합니다. <br>
> **공간 음향**이란 소리의 방향과 거리에 따라 소리가 어떻게 변하는지를 나타내는 것입니다. <br>
> **공간 음향**과 관련된 아래 세 가지 원리를 활용하여 <br>
> 사용자가 위치를 이동함에 따라 실시간으로 소리의 변화를 느낄 수 있도록 구현하였습니다.

1. **HRTF(Head-Related Transfer Function)** 를 활용하여 구현하였습니다.<br> 
**HRTF**는 소리가 귀에 도달할 때 거리를 고려하여 미세한 시간 차이를 수학적으로 모델링한 함수입니다.<br>
  <img src="https://github.com/user-attachments/assets/6e7c18ae-6d5a-404b-802d-cea38d465ccf">

- [x, y, z] 좌표에 따라 거리가 멀어지면 소리가 늦게 귀에 도달하기 때문에 볼륨을 **작게** 조절합니다.

- [x, y, z] 좌표에 따라 거리가 가까워지면 소리가 빨리 귀에 도달하기 때문에 볼륨을 **크게** 조절합니다.

✔️ 3D 공간에서의 스피커와 사용자 위치를 기반으로 소리의 **감쇠효과**를 구현하였습니다.

<br>

2. **IID(Interaural Intensity Difference)** 를 활용한 스피커의 수평적 움직임에 대한 소리의 전달입니다.<br> 
**IID**는 왼쪽에 위치한 스피커에서 전달된 음이 오른쪽 귀에 도달하기 전에 머리를 거치기 때문에<br>
왼쪽 귀에 전달되는 음파 에너지보다 적게 전달된다는 개념입니다.<br>
  <img src="https://github.com/user-attachments/assets/fe850ef5-f169-4151-b80b-33f99950a1b9">

- 사용자와 스피커 모델의 [x, z] 거리값을 기반으로 각도를 계산합니다.

- cos 함수를 활용하여 값이 `-1`에 가까우면 **왼쪽** 볼륨을, `1`에 가까우면 **오른쪽** 볼륨을 강조합니다.

  | 값  | 위치   | cos        |
  | --- | ------ | ---------- |
  | -1  | 왼쪽   | cos (180°) |
  | 0   | 중앙   | cos (90°)  |
  | 1   | 오른쪽 | cos (0°)   |

✔️ 스피커의 **수평(좌, 우)** 위치에 따른 공간 음향을 구현하였습니다.

<br>

3. 음파의 **흡수와 반사 작용**을 통한 스피커의 수직적 움직임에 대한 소리의 전달입니다.<br>
음파는 크게 파장이 짧은 고주파 음역대와 파장이 긴 저주파 음역대로 나눌 수 있습니다.<br>
파장의 길이에 의해 고주파 음역대는 **흡수와 반사**가 잘 되는 특징과<br>
저주파 음역대가 흡수, 반사가 잘 되지 않고 멀리 전달되는 특징을 활용하였습니다.
  <img src="https://github.com/user-attachments/assets/b425bd57-81ef-41b7-bc3e-9343752170ba">

- 스피커의 수직적 움직임을 천장과 바닥으로 제한합니다.

- 바닥에 있을 때에는 고주파 음파 에너지가 푹신한 바닥에 **흡수**되어 **저주파 음역대**를 강조합니다.

- 딱딱한 천장에 있을 때에는 고주파 음파 에너지가 **흡수**되기 보다는 **반사**되어 **고주파 음역대**를 강조합니다.

✔️ 스피커의 **수직** 위치에 따른 공간 음향을 구현하였습니다.

특히 스피커의 위치에 따라 음향 변화를 느끼는 것이 핵심 기술인만큼 사용자의 세밀한 위치 조작은 이 프로젝트에서 중요한 포인트입니다. <br>
마우스 조작만 허용했을 때, 아래와 같은 어려움을 발견할 수 있었습니다.

- 복잡한 3D 모델을 다루기 위해 오직 마우스로 모드를 전환하고 다시 모델을 조작하는 작업은 번거로울 수 있습니다.

#### (2) 게임에 스며들어 있는 멀티 인터페이스 (마우스, 키보드)

많은 게임들은 사용자가 빠른 반응속도로 조작할 수 있게 하기 위해 **키보드와 마우스를 함께** 사용하는 인터페이스를 제공합니다.<br> 
예를 들어, FPS 게임에서는 마우스로 시점을 조작하고 키보드로 캐릭터의 이동과 행동을 제어합니다.<br> 
이러한 배치는 한 손은 방향 조작, 다른 한 손은 액션을 수행하는 등 역할 분담을 나누어 입력을 **동시에** 수행하고 <br>
필요한 기능을 즉각적으로 사용할 수 있고 게임 플레이가 원활해지는 효과를 볼 수 있습니다.

또한 아래 과학적 연구에서 **양손 조작 인터페이스**의 이점이 입증되었습니다.

이정호 교수, 개인용 컴퓨터를 위한 멀티 터치 입력장치에 관한 연구, 2007, p.1-10<br><br><br>
_"데스크탑 환경에 기반을 둔 가상 환경에서 양손 상호작용은 작업 성능을 향상시키고 일의 효율을 증가시킬 수 있다."_<br><br>
K. Hinckley, Two-handed Virtual Manipulation Vol.5, No.3, 1998, p.260-302

_✔️ 위 분석을 통해 키보드와 마우스와 **양손 조작 인터페이스**를 구현하였습니다._

#### (3) 왼손 최적화된 키보드 배치

3D 모델을 드래그하여 실시간으로 음향의 변화를 구현하는 이 프로젝트에서는 사용자 경험을 개선하기 위해<br> 
키보드와 마우스를 동시에 활용하는 인터페이스를 채택하기로 결심했습니다.<br>
또한 아래와 같이 왼손 최적화된 키보드의 배치를 통해 왼손이 넓은 키보드 자판을 모두 조작해야 하는 번거로움을 해소해야 한다고 생각했습니다.

<img src="https://github.com/user-attachments/assets/ec3bd990-3318-465e-897d-2043b61764a3">

- 1, 2, 3 키: 사용자가 모드 (시점 이동 모드, 드래그 모드, 회전 모드)를 전환할 때 사용되며 왼손의 자연스러운 위치에 있습니다.

- Shift 키: 사용자가 스피커를 수직으로 이동할 때 사용합니다.

- Z, X 키: 사용자가 자동 저장된 위치 정보로 이동할 때 사용합니다.

_✔️ 사용자가 오른손으로 마우스를 조작하면서 왼손으로 키보드 입력을 **병행**할 수 있도록 구현하였습니다._

### 자동 저장: 소중한 위치 정보 관리하기

> 사용자가 최적의 스피커 위치 정보를 프리셋에 저장할 수 있지만 실수로 저장하지 않는다면 동일한 설정을 재현하기 어렵고,<br>
> 이는 작업 효율성과 사용자 경험을 저해할 수 있다고 생각했습니다.<br>
> 이러한 문제를 해결하고자 프리셋 저장 기능뿐만 아니라 **자동 저장** 기능을 구현하였습니다.

#### (1) 프리셋 저장과 자동 저장은 어떻게 다른걸까요?

| 프리셋 저장                                                                                               | 자동 저장                                                                                                 |
| --------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| <img width="400px" src="https://github.com/user-attachments/assets/bbdd2069-5c92-4ad8-b81c-647e91cb6212"> | <img width="400px" src="https://github.com/user-attachments/assets/272c4638-90a1-4e31-94ea-4550ff3e783f"> |

- 프리셋 저장: 사용자가 프리셋 저장 버튼을 클릭한 경우에만 위치 정보를 저장할 수 있습니다.

- 자동 저장: 사용자가 조작한 모델의 위치가 바뀐 경우 위치 정보를 **자동으로** 저장할 수 있습니다.

프리셋 저장과 자동 저장의 가장 큰 차이점은 저장 방식입니다.<br> 
프리셋 저장의 경우 사용자가 특정 동작 (버튼 클릭)을 수행해야 저장이 되는 반면, 자동 저장의 경우 모델의 위치가 변경됐을 때 자동으로 저장이 됩니다.<br>
자동으로 저장되는 방식은 사용자가 실수로 특정 동작을 하지 않았을 때 발생하는 불편함을 해소할 수 있다고 판단하였습니다.

#### (2) 위치 정보는 어디에서 관리하나요?

온라인 사용자의 경우 프리셋 저장과 자동 저장 기능 모두 **서버**에서 관리합니다.<br> 
웹을 사용하다 보면 사용자가 의도치 않게 페이지를 새로고침 하거나 브라우저를 종료하는 경우가 발생합니다.<br> 
이 때 위치 정보가 손실되기도 하는데 이 문제를 해결하기 위해 정보를 **관리할 방식과 장소**가 중요한 포인트라고 생각했습니다.<br>
이 프로젝트에서 위치 정보 저장과 관련된 **사용자 경험을 증진**시키기 위해 **서버**와 **로컬스토리지** 사이에서 아래와 같이 비교하고 결정하였습니다.

|      | 서버                                                  | 로컬스토리지                                   |
| ---- | ----------------------------------------------------- | ---------------------------------------------- |
| 장점 | 데이터 보존성 강화<br>다중 기기 동기화<br>보안성 향상 | 오프라인 사용 가능<br>빠른 접근 속도           |
| 단점 | 인터넷 연결 필요<br>서버 비용                         | 데이터 유실 위험<br>동기화 불가<br>보안 취약성 |

✔️ 위치 정보의 데이터 **보존**이 중요한 포인트인 만큼 로컬스토리지의 경우, 사용자가 브라우저의 데이터를 제거하는 상황, 브라우저의 저장 공간 정책에 따라 자동으로 삭제되는 상황을 고려하여 **서버**로 관리하는 것이 적합하다고 판단하였습니다.

#### (3) 만약 인터넷이 끊겼다면?

이 프로젝트에서는 오프라인 사용자를 위해 위치 정보 데이터를 **로컬스토리지**에서 관리합니다.<br>
또한 오프라인인 사용자가 온라인으로 전환되는 경우, 서버 데이터의 **정확성**을 유지하기 위해 <br>
**로컬스토리지**에서 임시로 저장된 데이터를 동기화하는 기능을 구현하였습니다.

<img src="https://github.com/user-attachments/assets/88ab59e6-3afb-40a9-bcdf-525c4c7b4273">

하지만 위와 같은 기능을 구현할 때 서버의 데이터가 로컬 스토리지에 저장된 데이터와 충돌이 발생합니다.<br> 
이 때 **낙관적 일치성(Optimistic Concurrency)** 이란 개념을 활용하여 해결하였는데 **낙관적 일치성**이란 사용자가 저장하는 데이터의 정확성을 유지하는 것을 의미합니다.<br> 
아래는 **낙관적 일치성** 개념을 활용한 구현 방식입니다.

- 데이터 베이스에 데이터는 버전 필드를 반드시 포함합니다.

- **로컬스토리지**에 저장된 위치 정보 데이터와 서버에 일치하는 데이터가 있는지 확인합니다.

  - 일치하는 데이터가 있다면, 업데이트를 거부합니다.

  - 일치하는 데이터가 없다면, 최신 저장된 버전 다음으로 업데이트를 진행합니다.

✔️ 데이터의 정확성을 유지하면서 **로컬스토리지**를 활용하여 오프라인, 온라인 사용자가 모두 사용할 수 있게끔 **사용자 접근성**을 향상시켰습니다.
