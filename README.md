### 1️⃣ 조작에 따라 달라지는 위치: 2D 화면에서 3D 조정하기

> 2D 화면인 모니터에서 3D 모델의 위치를 조작하려면 평면에서 **3D 좌표의 거리감**을 파악하는 것이 중요하다고 생각합니다. <br>
> 이 프로젝트에서는 변환된 2D 컴퓨터 화면 상의 좌표를 활용하여 거리감을 계산하고 위치를 파악하는 방식으로 구현하였습니다. <br>

1. 화면의 [x, y] 좌표를 **NDC(Normalized Device Coordinates)** 좌표로 변환합니다.<br>**NDC** 좌표란 화면의 모든 좌표를 `-1`에서 `1` 사이의 값으로 변환(정규화)한 좌표를 의미합니다.<br>
   <img src="https://github.com/user-attachments/assets/635258b9-4f44-436d-82e6-560064e78394">

<br>

2. 변환된 **NDC** 좌표를 기반으로, 카메라에서 시작해 마우스 위치를 향하는 **광선(Raycaster)** 을 생성합니다.<br>사용자가 마우스를 이동할 때 **광선**이 생성되어 아래 그림과 같이 좌표 이동을 감지합니다.<br>

### 2️⃣ 위치에 따라 달라지는 소리: 공간 음향 구현하기

> 스피커와 리스너의 위치에 따라 소리가 어떻게 들리는지를 구현하기 위해서 **공간 음향**이라는 개념이 중요하다고 생각합니다. <br> **공간 음향**이란 소리의 방향과 거리에 따라 소리가 어떻게 변하는지를 나타내는 것으로 이 프로젝트는 웹 브라우저에 내장된 **Web Audio API**를 활용하여 공간 음향과 관련된 세 가지 원리를 적용한 음향 변화를 구현하였습니다.<br>

1.  **HRTF (Head-Related Transfer Function)**
    <br>
    3D 공간 음향을 구현하기 위해 아래와 같이 **Web Audio API**에서 선택할 수 있는 두 가지 옵션이 있습니다.

    - **Equalpower**

      - 간단한 스테레오 효과를 구현하기 위해 사용되며 주로 2D 오디오 구현에 사용됩니다.

      - 좌, 우 채널 간의 볼륨 변화만 구현이 가능합니다.

    - **HRTF**

      - HRTF란 사용자의 머리를 중심으로 3D 공간에서의 음향 전달 방식을 수학적으로 모델링한 함수입니다.

      - 게임, 가상 현실에서 공간 음향을 구현하기 위해 사용되며 주로 **3D** 오디오 구현에 사용됩니다.

      - 위, 아래, 앞, 뒤 등 **다양한 방향**에서의 볼륨 변화 구현이 가능합니다.

    3D 공간 시뮬레이터에서 음향 변화를 구현하려면 **HRTF**가 적합하다고 결정하였습니다. 다양한 방향에서 음향 변화를 전달할 수 있는 **HRTF** 모델은 3D 공간에서의 스피커와 리스너 사이의 `[x, y, z]` 좌표 거리 차이를 기반으로 **소리의 감쇠효과**를 구현합니다. 아래와 같이 리스너로부터 스피커의 좌표 거리가 멀어지면 소리가 늦게 귀에 도달하기 때문에 볼륨을 **작게** 조절하고 가까워지면 소리가 빨리 귀에 도달하기 때문에 볼륨을 **크게** 조절합니다.
    <br>

    <img width="890" alt="스크린샷 2024-10-18 오후 9 57 01" src="https://github.com/user-attachments/assets/7ac13ed6-47f5-417f-b823-62bfc670a5be">

<br>

2. **IID(Interaural Intensity Difference)**<br>
   3D 공간 음향에서 소리의 좌, 우 방향성에 대한 미묘한 차이를 효과적으로 전달하기 위해서는 HRTF 뿐만 아니라 IID의 적용이 필요하다고 생각했습니다. **IID** 현상은 소리가 두 귀에 도달할 때의 음파의 에너지 수준 차이를 나타냅니다. 예를 들어, 스피커가 왼쪽에 위치하면 왼쪽 귀에는 직접적인 소리가 더 강하게 도달하고, 오른쪽 귀에는 머리가 막고 있어 소리가 약해지는 상황을 의미합니다.

   이 때, 리스너로부터 스피커의 왼쪽, 오른쪽 위치를 파악하기 위해 삼각함수 중 하나인 **cos** 함수를 사용하였습니다. **cos** 함수를 통해 스피커와 리스너의 x축, z축의 거리 차이로 각도를 계산하고 좌, 우에 관련된 값을 설정하였습니다. 아래와 같이 값이 `-1`에 가까우면 **왼쪽** 볼륨을, `1`에 가까우면 **오른쪽** 볼륨을 강조하였습니다.

   | 값  | 위치   | cos        |
   | --- | ------ | ---------- |
   | -1  | 왼쪽   | cos (180°) |
   | 0   | 중앙   | cos (90°)  |
   | 1   | 오른쪽 | cos (0°)   |

   <br>

    <img alt="스크린샷 2024-10-18 오후 10 19 11" src="https://github.com/user-attachments/assets/21f0408f-0381-4202-ad25-4f68daa72d6c">

<br>

3.  **음파의 흡수, 반사 작용** <br>
    3D 오디오 환경을 효과적으로 구현하기 위해, 단순한 볼륨 감쇠효과를 넘어서 음파의 흡수와 반사 원리를 활용하여 고주파와 저주파 음역대의 차이를 명확하게 조절하는 접근을 선택했습니다. 고주파 음역대와 저주파 음역대의 아래와 같이 특징을 활용하면 사용자에게 보다 입체적인 음향 변화를 제공할 수 있다고 판단하였습니다.

    - 고주파 음역대

      - 파장이 짧아 다양한 표면에서 쉽게 흡수되거나 반사됩니다.

      - 세밀한 소리의 질감과 **선명도**를 제공합니다.

    - 저주파 음역대

      - 파장이 길어 흡수와 반사가 적게 일어나며, 소리가 더 멀리까지 전달됩니다.

      - **깊고 울림**이 있는 소리를 강조합니다.

    또한, 스피커의 수직적 움직임을 천장과 바닥으로 제한하여 사용자에게 **직관적인 음향 변화**를 제공할 수 있도록 구현하였습니다.

    - 천장 위치의 스피커

      - 딱딱한 천장 표면은 고주파 음파의 반사를 증가시켜, 고주파 음역대가 더 강조됩니다.

### 1️⃣ 키보드 단축키: 혼자 바쁜 마우스 도와주기

#### (1) 마우스가 홀로 감당하기에 부담되는 작업들

마우스 하나로 모든 작업을 수행하려고 하는 것은 사용자 경험을 저해할 수 있다고 생각합니다. 특히 스피커의 위치에 따라 음향 변화를 느끼는 것이 핵심 기술인만큼 사용자의 **세밀한 위치 조정**은 중요하다고 생각합니다. 마우스 조작만 허용했을 때, 아래와 같은 어려움을 발견할 수 있었습니다.

- 복잡한 3D 모델을 다루기 위해 오직 마우스로 **모드를 전환**하고 다시 **모델을 조정**하는 작업은 번거로울 수 있습니다.<br>

#### (2) 게임에 스며들어 있는 멀티 인터페이스 (마우스, 키보드)

많은 게임들은 사용자가 빠른 반응속도로 조작할 수 있게 하기 위해 **키보드와 마우스를 함께** 사용하는 인터페이스를 제공합니다. 예를 들어, FPS 게임에서는 마우스로 시점을 조작하고 키보드로 캐릭터의 이동과 행동을 제어합니다. 이러한 배치는 한 손은 방향 조작, 다른 한 손은 액션을 수행하는 등 역할 분담을 나누어 입력을 **동시에** 수행하고 필요한 기능을 즉각적으로 사용할 수 있고 게임 플레이가 원활해지는 효과를 볼 수 있습니다.또한 아래 과학적 연구에서 **양손 조작 인터페이스**의 이점이 입증되었습니다.

_"데스크탑 환경에 기반을 둔 가상 환경에서 양손 상호작용은 작업 성능을 향상시키고 일의 효율을 증가시킬 수 있다."_<br><br>
K. Hinckley, Two-handed Virtual Manipulation Vol.5, No.3, 1998, p.260-302

#### (3) 왼손 최적화된 키보드 배치

3D 모델을 드래그하여 실시간으로 음향의 변화를 구현하는 이 프로젝트에서는 사용자 경험을 개선하기 위해 키보드와 마우스를 동시에 활용하는 인터페이스를 채택하기로 결심했습니다. 또한 **왼손 최적화**된 키보드의 배치를 통해 왼손이 넓은 키보드 자판을 모두 조작해야 하는 번거로움을 해소해야 한다고 생각했습니다.

### 2️⃣ 자동 저장: 사용자의 최적 위치 정보 관리하기

> 사용자가 최적의 스피커 위치 정보를 프리셋에 저장할 수 있지만 실수로 저장하지 않는다면 동일한 설정을 재현하기 어렵고,<br>
> 이는 작업 효율성과 사용자 경험을 저해할 수 있다고 생각했습니다.<br>
> 이러한 문제를 해결하고자 프리셋 저장 기능뿐만 아니라 **자동 저장** 기능을 구현하였습니다.

<br>

#### (1) 프리셋 저장과 자동 저장은 어떻게 다른걸까요?

| 프리셋 저장                                                                                               | 자동 저장                                                                                                 |
| --------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| <img width="400px" src="https://github.com/user-attachments/assets/bbdd2069-5c92-4ad8-b81c-647e91cb6212"> | <img width="400px" src="https://github.com/user-attachments/assets/272c4638-90a1-4e31-94ea-4550ff3e783f"> |

- 프리셋 저장: 사용자가 프리셋 저장 **버튼을 클릭**한 경우에만 위치 정보를 저장할 수 있습니다.

- 자동 저장: 사용자가 조작한 모델의 위치가 바뀐 경우 위치 정보를 **자동으로** 저장할 수 있습니다.

프리셋 저장과 자동 저장의 가장 큰 차이점은 저장 방식입니다. 프리셋 저장의 경우 사용자가 **버튼을 클릭**해야 저장이 되는 반면 자동 저장의 경우 모델의 위치가 변경됐을 때 **자동으로** 저장이 됩니다. 자동으로 저장되는 방식은 사용자가 실수로 저장 버튼을 클릭하지 않았을 때 발생하는 불편함을 해소할 수 있다고 판단하였습니다.

<br>

#### (2) 사용자가 조정한 스피커와 리스너의 위치는 어디에 저장하나요?

온라인 사용자의 경우 프리셋 저장과 자동 저장 기능 모두 **서버**에서 관리합니다. 웹을 사용하다 보면 사용자가 의도치 않게 페이지를 새로고침 하거나 브라우저를 종료하는 경우가 발생합니다. 이 때 위치 정보가 손실되기도 하는데 이 문제를 해결하기 위해 정보를 **관리할 방식과 장소**가 중요한 포인트라고 생각했습니다. 위치 정보 저장과 관련된 **사용자 경험을 증진**시키기 위해 아래와 같이 **서버**와 **로컬스토리지**를 비교하고 결정하였습니다.

|      | 서버                                                  | 로컬스토리지                                   |
| ---- | ----------------------------------------------------- | ---------------------------------------------- |
| 장점 | 데이터 보존성 강화<br>다중 기기 동기화<br>보안성 향상 | 오프라인 사용 가능<br>빠른 접근 속도           |
| 단점 | 인터넷 연결 필요<br>서버 비용                         | 데이터 유실 위험<br>동기화 불가<br>보안 취약성 |

✔️ 위치 정보의 데이터 **보존**이 중요한 포인트인 만큼 **로컬스토리지**의 경우에는 예외 상황이 발생합니다.

- 사용자가 브라우저의 데이터를 제거하는 상황
- 브라우저의 저장 공간 정책에 따라 자동으로 삭제되는 상황

이 점을 고려하여 **서버**로 관리하는 것이 적합하다고 판단하였습니다.

<br>

#### (3) 로그인을 하지 않은 사용자가 갑자기 로그인을 시도한다면?

이 프로젝트에서는 오프라인 사용자를 위해 위치 정보 데이터를 **로컬스토리지**에서 관리합니다. 또한 오프라인인 사용자가 온라인으로 전환되는 경우, 서버 데이터의 **정확성**을 유지하기 위해 **로컬스토리지**에서 임시로 저장된 데이터를 동기화하는 기능을 구현하였습니다.

<img src="https://github.com/user-attachments/assets/88ab59e6-3afb-40a9-bcdf-525c4c7b4273">

하지만 위와 같은 기능을 구현할 때 서버의 데이터가 로컬 스토리지에 저장된 데이터와 충돌이 발생합니다.<br>
이러한 상황은 **낙관적 일치성(Optimistic Concurrency)** 이란 개념을 활용하여 해결하였습니다.<br>
**낙관적 일치성**이란 사용자가 저장하는 데이터의 정확성을 유지하는 것을 의미합니다.<br>
아래는 **낙관적 일치성** 개념을 적용한 구현 방식입니다.

- 데이터 베이스에 저장되는 데이터는 **버전** 필드를 반드시 포함합니다.

- **로컬스토리지**에 저장된 위치 정보 데이터와 서버에 일치하는 데이터가 있는지 확인합니다.

  - 일치하는 데이터가 있다면, 업데이트를 거부합니다.

  - 일치하는 데이터가 없다면, 최신 저장된 다음 **버전**으로 업데이트를 진행합니다.

✔️ 데이터의 정확성을 유지하면서 **로컬스토리지**를 활용하여 오프라인, 온라인 사용자가 모두 사용할 수 있게끔 **사용자 접근성**을 향상시켰습니다.
